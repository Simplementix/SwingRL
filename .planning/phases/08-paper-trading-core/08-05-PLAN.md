  <action>
    Read existing Dockerfile first. Modify it to add multi-stage build:

    **Dockerfile changes:**
    - Keep existing build as a `ci` stage (rename FROM ... AS ci) that includes dev deps (pytest, ruff, mypy, bandit). This is what ci-homelab.sh uses.
    - Add `production` stage that:
      1. FROM python:3.11-slim AS production
      2. Same non-root trader user (UID 1000) pattern
      3. Install only main dependencies (uv sync --no-dev or equivalent without --group dev)
      4. COPY src/, config/, scripts/ into image
      5. Do NOT copy tests/, docs/, .planning/
      6. Set HEALTHCHECK: `HEALTHCHECK --interval=60s --timeout=10s --retries=3 CMD ["python", "scripts/healthcheck.py"]`
      7. ENTRYPOINT or CMD that starts the main application process (for Phase 8: just keep container alive for manual cycle triggers. Phase 9 will add APScheduler).
      8. For Phase 8, use: `CMD ["python", "-c", "import time; time.sleep(999999)"]` as a placeholder. In Phase 9, this becomes the APScheduler entry point. Alternatively, use `CMD ["tail", "-f", "/dev/null"]` for a simpler idle container. Document this is a placeholder.
    - Ensure ci-homelab.sh still works by targeting the `ci` stage: `docker build --target ci ...`

    **scripts/healthcheck.py:**
    - Lightweight probe that:
      1. Checks if the main Python process is alive (os.getpid() exists -- trivially true if this script runs)
      2. Opens SQLite trading_ops.db and runs `PRAGMA integrity_check` (verifies DB accessible)
      3. Opens DuckDB market_data.ddb and runs `SELECT 1` (verifies DuckDB accessible)
      4. Exits 0 on success, 1 on any failure
    - Use pathlib for all paths
    - Use config/swingrl.yaml to discover db paths (or hardcode db/ since it's the default)
    - Should complete in less than 5 seconds

    **docker-compose.prod.yml:**
    - Based on existing docker-compose.yml structure
    - Service name: swingrl
    - Build with target: production
    - Environment: SWINGRL_TRADING_MODE=paper, TZ=America/New_York
    - Volumes: bind mount data/, db/, models/, logs/, config/ (same as existing)
    - env_file: .env (for ALPACA_API_KEY, ALPACA_SECRET_KEY, DISCORD_WEBHOOK_URL)
    - Resource limits: 2.5g mem, 1 CPU (same as existing)
    - restart: unless-stopped
    - No ports exposed (no web interface)
    - Comment documenting this is for homelab paper trading deployment

    **Update ci-homelab.sh if needed:**
    - Verify it uses `--target ci` for builds. If not, add it so it doesn't accidentally use the production stage.

    Verify the production image builds:
    ```bash
    docker build --target production -t swingrl:paper-test .
    docker build --target ci -t swingrl:ci-test .
    ```
