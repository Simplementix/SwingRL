## Implementation Decisions

### Phase Boundary Clarifications
- No nomic-embed-text — SQL filtering + LLM reasoning is sufficient (Phase 19 decision upheld)
- Training-side infrastructure + API service only — no live enrichment modules
- Full training API endpoints: /ingest, /consolidate, /health, /training/run_config, /training/epoch_advice
- All DuckDB schema migrations in scope (training_epochs, meta_decisions, reward_adjustments + ALTER training_runs)
- Re-run seed_memory_from_backtest.py against live memory service
- Update ROADMAP.md phase goal to match these decisions
- 6 training iterations (1 baseline + 5 memory-enhanced) are a phase completion requirement

### swingrl-memory Service Architecture
- Code lives inside SwingRL repo: services/memory/
- Separate modules per agent: services/memory/agents/ingest.py, consolidate.py, query.py
- Own Docker container with separate Dockerfile (services/memory/Dockerfile)
- Own requirements.txt: fastapi, uvicorn, httpx (for async Ollama calls)
- httpx async for all Ollama API calls (non-blocking, natural FastAPI fit)
- SQLite memory.db stored in db/ directory (alongside trading_ops.db and market_data.ddb)
- Consolidation scheduler runs inside the memory service (30-min background schedule)
- Service blocks on startup until Ollama is healthy (wait, don't start degraded)
- Full training API: /training/run_config returns hyperparams + reward weights + curriculum windows
- Shared ruff/mypy config from repo root (line-length=100, snake_case, same standards)
- Tests in main tests/ directory (tests/test_memory_service.py pattern)

### Docker Stack Layout
- **Ollama container:** Internal DNS only (no host port). Named Docker volume 'ollama_models' for persistence. 24GB RAM / 8 CPUs limit
- **swingrl-memory container:** Port 8889 exposed to host (needed for validation scripts). Simple single-stage Dockerfile (python:3.11-slim)
- **Model pull strategy:** Init script (scripts/setup_ollama.sh). CI pulls if missing in volume
- **Startup order:** depends_on + healthcheck: Ollama (curl /api/tags) -> swingrl-memory -> swingrl
- **SAC buffer_size:** Default 500K (configurable via SwingRLConfig). Proven 200K works; 500K fits with 24GB Ollama allocation

### Training Re-run Scope
- Full re-run: all 6 algo x env combos (PPO/A2C/SAC x equity/crypto), sequential (equity first, then crypto)
- 1 baseline run (no memory) + 5 memory-enhanced runs = 6 total iterations
- Automated back-to-back: train_pipeline.py --iterations 5 (baseline is separate). Operator walks away
- Same timesteps as Phase 19: 1M equity, 500K crypto with escalation to 2M/1M if Sharpe < 0.5
- Same walk-forward folds for A/B comparison (first iteration). Memory runs can use curriculum sampler after cold-start
- Resumable iterations: state tracked in data/training_state.json. Survives power outages
- Best per-algo deployment: across all 6 runs, deploy whichever iteration produced best Sortino per algo x env. Calmar as tiebreaker
- Auto-generated comparison report: data/training_comparison.json with per-algo metrics across all runs
- All testing on homelab — M1 Mac cannot handle memory agents. Mocked unit tests run locally, integration on homelab

### Cold-Start Behavior
- Baseline run counts toward cold-start threshold (3 completed runs in DuckDB)
- Only successfully completed runs count (WHERE status = 'completed')
- Per algo x env granularity (each PPO-equity, SAC-crypto, etc. = 1 run)
- Seeded backtest data provides memory context but does NOT count as completed runs

### Failure Recovery
- Ollama/memory service failure: indefinite pause with 30s health polling + Discord alert (first alert immediate, then every 15 min)
- Training resumes automatically when service is back
- Same pause behavior for swingrl-memory FastAPI crashes
- Algo failure: retry once, then skip to next algo. Mark failed in DuckDB + Discord alert
- Power outage: checkpoint resume via model.zip existence + iteration resume via data/training_state.json

### Monitoring During Runs
- Discord alerts (via existing config.discord.webhook_url, no-op if unconfigured):
  - Iteration start/end with summary metrics
  - Per-algo completion: Sharpe, MDD, Sortino, runtime
  - Gate pass/fail
  - Memory agent events: meta-trainer activation, early stop, reward weight adjustments
  - Ollama resource stats (RAM, tokens/sec, loaded models) included in all embeds
- Final comparison embed after all 6 iterations with full metrics table and learning progression

### CI Pipeline Changes
- ci-homelab.sh: mock Ollama in CI (existing mocked MemoryClient). No real Ollama in CI
- CI builds and lints services/memory/ code (ruff + mypy)
- CI pulls Ollama models if missing in volume (first CI run only)
- Simple single-stage Dockerfile for memory service

### Environmental Context for Memory (9 Source Tags)
- 5 existing training tags: training_run:historical, training_epoch:historical, reward_adjustment:historical, backtest_fill:historical, curriculum_performance:historical
- 4 NEW context tags:
  - macro_correlation:historical — FRED values (VIX, DFF, T10Y2Y, CPI, UNRATE) + fold performance for both equity and crypto
  - regime_transition:historical — HMM regime changes with timestamps and macro context
  - cross_asset:historical — Equity vs crypto performance per fold with macro overlay
  - feature_importance:historical — Top-5 most influential features per algo per regime
- Raw TA indicators stay with RL agents (already in 156-dim equity / 45-dim crypto obs vectors)
- Memory captures meta-patterns about WHEN and WHY features/regimes drive performance

### Ollama System Prompts
- Domain-specific prompts: include SwingRL context (asset classes, capital preservation, regime types, metric definitions)
- Hardcoded as Python string constants in each agent module (version-controlled)
- Tiered retrieval: consolidated patterns first, raw memories on demand
- Tool calling for structured output (qwen3:14b supports it). Fallback: retry with JSON mode, then return defaults
- Conflict detection: ConsolidateAgent flags contradicting consolidations with 'conflicting_with' field

### Memory Consolidation Quality
- Structured output schema required: specific pattern with numbers, affected algos/envs, actionable implication
- Rejected if missing required fields. Retry once, then discard. Raw memories stay for next cycle
- Consolidation quality logged to DuckDB (consolidation_quality table): attempt_count, accepted, rejected_reason
- Conflict detection: flag contradicting consolidations for QueryAgent to present both perspectives

### Memory Growth & Debug
- No pruning between iterations — consolidation handles compression, 90-day archival for long-term
- Both background schedule (30-min) AND explicit POST /consolidate between iterations
- Debug endpoints: GET /memories (filter by source, limit, date range) and GET /consolidations
- Read-only, for operator inspection and validation

### Security
- Minimal hardening: XML/delimiter wrapping around memory text in prompts. bounds.py clamps all LLM outputs
- API key header (X-API-Key) on all endpoints. Key stored in .env
- Network: Ollama internal-only, memory service exposed on localhost:8889

<specifics>
## Specific Ideas

- Google's Always-On Memory Agent pattern adapted: 3 focused agents (Ingest, Consolidate, Query), each with own system prompt
- "Sleep consolidation" metaphor — consolidation finds cross-run patterns that individual runs don't reveal
- Two-layer knowledge: DuckDB for hard numbers (exact Sharpe, MDD), memory.db for soft knowledge (patterns, causal insights)
- Operator should be able to walk away and check results in 5-7 days — fully hands-off training
- Memory agents should enter live trading "educated" about market dynamics (macro correlations, regime patterns) — no cold start for live trading
- The 5 iterative memory runs demonstrate learning-over-time: each run should produce progressively better (or at least different) LLM advice
- Capital preservation is the primary constraint — Sortino (upside-focused) as primary ranking metric, Calmar (return-per-drawdown) as tiebreaker

</specifics>
