## Summary

Phase 19.1 builds the full memory agent stack: the swingrl-memory FastAPI service (services/memory/), Ollama as a Docker sidecar with Qwen models, DuckDB schema migrations for training telemetry, MetaTrainingOrchestrator wiring into train_pipeline.py, and 6 sequential training iterations with best-model deployment.

The codebase already has all the client-side pieces: MemoryClient (fail-open urllib), MetaTrainingOrchestrator (outer loop), bounds.py (safety clamps), MemoryVecRewardWrapper, MemoryEpochCallback, MemoryCurriculumSampler, and validation scripts. What is missing is the server-side service (services/memory/), the Docker composition changes, the DuckDB DDL, and the multi-iteration orchestration layer in train_pipeline.py.

This phase is primarily about integration engineering — connecting existing pieces — not algorithm design. The hardest parts are the multi-iteration training loop with resumable state, the Ollama startup health loop, and the per-iteration model selection logic (best Sortino, Calmar tiebreak).

**Primary recommendation:** Build services/memory/ as a plain FastAPI app with three agent modules and a SQLite backend. Wire the MetaTrainingOrchestrator into train_pipeline.py behind a `--iterations N` flag. Track iteration state in data/training_state.json with atomic writes.

### Pattern 1: FastAPI Lifespan with APScheduler
**What:** Use FastAPI's lifespan context manager (replaces deprecated on_event) to start/stop the 30-min consolidation scheduler.
**When to use:** All background scheduling in FastAPI as of 2024+.
**Example:**
```python
# Source: FastAPI docs + APScheduler 3.x pattern
from contextlib import asynccontextmanager
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from fastapi import FastAPI

scheduler = AsyncIOScheduler()

@asynccontextmanager
async def lifespan(app: FastAPI):
    # Startup: wait for Ollama health, then start scheduler
    await _wait_for_ollama()
    scheduler.add_job(run_consolidation, "interval", minutes=30)
    scheduler.start()
    yield
    # Shutdown
    scheduler.shutdown()

app = FastAPI(lifespan=lifespan)
```

### Pattern 2: API Key Header Dependency
**What:** FastAPI Security dependency injected per-router. Validates X-API-Key header against env var.
**When to use:** All endpoints except /health (which is unauthenticated for Docker healthcheck).
**Example:**
```python
# Source: FastAPI docs + project pattern
from fastapi import Security, HTTPException, status
from fastapi.security import APIKeyHeader

api_key_header = APIKeyHeader(name="X-API-Key", auto_error=False)

def verify_api_key(key: str | None = Security(api_key_header)) -> str:
    """Validate X-API-Key header against MEMORY_API_KEY env var."""
    expected = os.environ.get("MEMORY_API_KEY", "")
    if not expected or key != expected:
        raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED)
    return key
```

### Pattern 3: Ollama Structured Output (httpx async)
**What:** POST to Ollama /api/chat with `format` JSON schema for deterministic structured responses. qwen3:14b supports tool calling as first option; fallback to format-constrained JSON.
**When to use:** /training/run_config and /training/epoch_advice — where LLM must return typed dicts.
**Example:**
```python
# Source: https://docs.ollama.com/capabilities/structured-outputs
import httpx

async def _call_ollama_structured(
    prompt: str,
    schema: dict,
    model: str = "qwen3:14b",
    ollama_url: str = "http://swingrl-ollama:11434",
) -> dict:
    """Call Ollama with JSON schema format for structured output."""
    async with httpx.AsyncClient(timeout=30.0) as client:
        resp = await client.post(
            f"{ollama_url}/api/chat",
            json={
                "model": model,
                "messages": [{"role": "user", "content": prompt}],
                "format": schema,
                "stream": False,
                "options": {"temperature": 0},
            },
        )
        resp.raise_for_status()
        body = resp.json()
        import json
        return json.loads(body["message"]["content"])
```

### Pattern 4: Ollama Startup Health Loop
**What:** Block service startup until Ollama /api/tags returns 200. Use asyncio.sleep loop with timeout. Never start degraded.
**When to use:** services/memory/main.py lifespan startup, before scheduler starts.
**Example:**
```python
# Source: Ollama docs + project CONTEXT.md decision
import asyncio
import httpx

async def _wait_for_ollama(
    url: str = "http://swingrl-ollama:11434",
    timeout_sec: int = 300,
) -> None:
    """Block until Ollama /api/tags returns 200. Raises on timeout."""
    deadline = asyncio.get_event_loop().time() + timeout_sec
    while asyncio.get_event_loop().time() < deadline:
        try:
            async with httpx.AsyncClient(timeout=5.0) as client:
                resp = await client.get(f"{url}/api/tags")
                if resp.status_code == 200:
                    return
        except Exception:
            pass
        await asyncio.sleep(5)
    raise RuntimeError(f"Ollama not healthy after {timeout_sec}s")
```

### Pattern 5: Atomic Training State File
**What:** data/training_state.json written atomically (write to .tmp then os.replace) to survive power outages.
**When to use:** Before and after every training iteration in train_pipeline.py.
**Example:**
```python
import json
import os
from pathlib import Path

def save_training_state(state: dict, path: Path) -> None:
    """Atomically write training state to JSON file."""
    tmp = path.with_suffix(".tmp")
    tmp.write_text(json.dumps(state, indent=2, default=str))
    os.replace(tmp, path)  # atomic on POSIX

def load_training_state(path: Path) -> dict:
    """Load training state, returning empty dict if file missing."""
    if not path.exists():
        return {"completed_iterations": [], "current_iteration": 0}
    return json.loads(path.read_text())
```

### Pattern 7: Multi-Iteration Training Loop
**What:** train_pipeline.py gains --iterations N flag. Iteration 0 is baseline (memory disabled). Iterations 1-N enable memory. State persisted to data/training_state.json between iterations.
**When to use:** `python scripts/train_pipeline.py --env all --iterations 5`
**Example:**
```python
# Pseudocode for iteration loop in train_pipeline.py
def run_all_iterations(
    base_config: SwingRLConfig,
    iterations: int,
    state_path: Path,
    report_path: Path,
) -> None:
    state = load_training_state(state_path)

    for i in range(iterations + 1):  # 0 = baseline, 1..N = memory
        if i in state["completed_iterations"]:
            log.info("iteration_skipped_checkpointed", iteration=i)
            continue

        # Iteration 0: memory disabled (baseline)
        cfg = base_config.model_copy(deep=True)
        if i == 0:
            cfg.memory_agent.enabled = False
            cfg.memory_agent.meta_training = False
        else:
            cfg.memory_agent.enabled = True
            cfg.memory_agent.meta_training = True

        state["current_iteration"] = i
        save_training_state(state, state_path)

        # Run full pipeline (both envs sequential)
        result = run_environment("equity", cfg, ...)
        run_environment("crypto", cfg, ...)

        state["completed_iterations"].append(i)
        state[f"iteration_{i}_result"] = result
        save_training_state(state, state_path)

        # Consolidate memories between iterations (explicit POST)
        if i > 0:
            memory_client.consolidate(timeout=120.0)

    # Select best per algo x env (Sortino rank, Calmar tiebreak)
    _deploy_best_models(state, models_dir)
    _write_comparison_report(state, report_path)
```

### Anti-Patterns to Avoid
- **Starting swingrl-memory before Ollama is healthy:** The service MUST block startup until Ollama responds. Starting degraded creates misleading "healthy" service that fails all LLM calls silently.
- **Relative imports in services/memory/:** This is a standalone service — absolute imports within its package are fine, but do not try to import from src/swingrl/ (different container context).
- **Holding DuckDB connection across iterations:** Project decision [Phase 19-02] — open/close per algo unit. Apply same pattern in iteration loop.
- **Blocking asyncio event loop on Ollama calls:** Use httpx async client everywhere in the FastAPI service. Never urllib inside async routes.
- **Writing training_state.json non-atomically:** Use os.replace(tmp, final) — plain write_text risks corruption on power loss.
- **Embedding API key in Docker image:** Store MEMORY_API_KEY in .env only, never baked into image layer.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Structured LLM output | Custom parsing/regex | Ollama format= JSON schema field | Ollama handles schema enforcement; fallback is retry-then-defaults pattern |
| Background scheduling | Custom threading.Timer loop | APScheduler AsyncIOScheduler | APScheduler integrates with FastAPI lifespan; handles missed jobs, graceful shutdown |
| API key auth | Custom header parsing middleware | FastAPI APIKeyHeader Security dependency | Integrates with OpenAPI docs, handles 401 automatically |
| Memory deduplication | Custom hash comparison | SQL WHERE + timestamp: recent-first query | Simple SQL is sufficient; no vector index needed (no embeddings by design) |
| HTTP retry logic for Ollama | Manual retry loop | httpx built-in + tenacity | tenacity already in pyproject.toml; use @retry decorator on agent calls |
| Docker health polling | Custom shell script | Docker compose healthcheck + depends_on | Native compose feature handles retry/backoff |

**Key insight:** The service is a thin wrapper over SQLite + Ollama. The domain complexity is in the system prompts and clamping logic (already in bounds.py) — not in the infrastructure.

## Common Pitfalls

### Pitfall 1: Ollama curl Missing from Docker Image
**What goes wrong:** Docker healthcheck `CMD curl -f http://localhost:11434/api/tags` fails immediately — curl is not in the ollama/ollama image.
**Why it happens:** Ollama image is Alpine/minimal base without curl installed.
**How to avoid:** Use wget instead: `["CMD-SHELL", "wget -q --spider http://localhost:11434/api/tags || exit 1"]`. wget IS in the Ollama image.
**Warning signs:** Container shows "unhealthy" immediately despite Ollama being fine.

### Pitfall 2: APScheduler version incompatibility
**What goes wrong:** APScheduler 4.x has a completely different API (no BackgroundScheduler, different job syntax). Code written for 3.x silently fails or throws AttributeError.
**Why it happens:** pip/uv resolves to latest (4.x) unless pinned.
**How to avoid:** Pin `apscheduler>=3.10,<4` in services/memory/requirements.txt. This is a known project decision from STATE.md.
**Warning signs:** `AttributeError: 'AsyncIOScheduler' object has no attribute 'add_job'` or import errors.

### Pitfall 3: Cold-Start Guard Timing
**What goes wrong:** Memory meta-trainer is passive for all 6 iterations if baseline run doesn't write to DuckDB training_runs table with status='completed'.
**Why it happens:** MetaTrainingOrchestrator._get_run_history() queries `WHERE run_type = 'completed'` — not `status`. The column is `run_type` not `status`. AND the baseline run (iteration 0) must write completed rows even though memory is disabled.
**How to avoid:** Baseline run uses direct TrainingOrchestrator.train() but the pipeline must still write DuckDB training_runs rows with `run_type='completed'`. Verify in _write_model_metadata() that run_type is set.
**Warning signs:** All 5 memory iterations show `meta_cold_start_guard` in logs (completed_runs=0).

### Pitfall 4: services/memory/ can't import src/swingrl/
**What goes wrong:** services/memory/ code attempts `from swingrl.config.schema import ...` — this works on the developer's laptop (PYTHONPATH=src) but fails inside the swingrl-memory container which doesn't have src/swingrl/ mounted.
**Why it happens:** services/memory/ is an independent service — it has its own requirements.txt and does NOT get the SwingRL package installed.
**How to avoid:** services/memory/ must be entirely self-contained. Config (API key, Ollama URL, DB paths) comes from environment variables, not SwingRLConfig. The SQLite db path is passed via env var MEMORY_DB_PATH.
**Warning signs:** ImportError on container startup for any swingrl.* module.

### Pitfall 5: DuckDB Write-Ahead Log Contention
**What goes wrong:** training_pipeline.py opens DuckDB read-write while swingrl-memory service also queries it (for cold-start guard). DuckDB only allows one writer — concurrent writes from two processes causes "database is locked" error.
**Why it happens:** DuckDB supports multiple readers OR one writer. In the training loop, the pipeline has a write connection open while memory service tries to read.
**How to avoid:** MetaTrainingOrchestrator opens DuckDB as `read_only=True` (already correct in existing code). The memory service must ALSO open read_only=True for cold-start guard queries. Write operations during training go through the pipeline only.
**Warning signs:** `IOException: database is locked` in memory service logs during training.

### Pitfall 6: Training State JSON Corruption on Interrupt
**What goes wrong:** Power outage or SIGKILL during json.dumps() write corrupts training_state.json — next startup reads partial JSON and crashes.
**Why it happens:** Plain Path.write_text() is not atomic.
**How to avoid:** Write to .tmp then os.replace() (POSIX atomic rename). Already documented in Pattern 5.
**Warning signs:** `json.JSONDecodeError` on startup for training_state.json.

### Pitfall 7: model.zip checkpoint detection vs. iteration selection
**What goes wrong:** Existing `_check_algo_checkpoint()` skips re-training if model.zip exists. In multi-iteration mode, iteration 2 would be skipped because iteration 1 wrote model.zip.
**Why it happens:** Checkpoint logic was designed for single-run resume, not multi-iteration.
**How to avoid:** Each iteration saves models to a versioned subdirectory: `models/iterations/iter_{N}/{env}/{algo}/model.zip`. The final "deploy best" step copies the winner to `models/active/`. The existing `--force` flag bypasses checkpoint but destroys previous run data.
**Warning signs:** All 5 memory iterations are "checkpointed" and skipped, generating no per-iteration metrics.

### Iteration Selection Logic (Best Sortino)
```python
# In train_pipeline.py or scripts/select_best_models.py
# Source: CONTEXT.md decision on Sortino rank, Calmar tiebreak

def select_best_per_algo_env(
    state: dict,
    models_dir: Path,
) -> dict[str, dict[str, int]]:
    """Select best iteration per algo x env by Sortino, Calmar tiebreak.

    Returns:
        Dict[env_name][algo_name] = best_iteration_index
    """
    best: dict[str, dict[str, dict]] = {}  # env -> algo -> {iter, sortino, calmar}

    for iter_idx, iter_data in state.items():
        if not str(iter_idx).startswith("iteration_"):
            continue
        idx = int(str(iter_idx).split("_")[1])
        for env_name in ["equity", "crypto"]:
            env_result = iter_data.get(env_name, {})
            for algo_name in ["ppo", "a2c", "sac"]:
                ft = env_result.get("final_training", {}).get(algo_name, {})
                sortino = ft.get("sortino", 0.0)
                calmar = ft.get("calmar", 0.0)
                current_best = best.setdefault(env_name, {}).get(algo_name)
                if current_best is None or sortino > current_best["sortino"] or (
                    sortino == current_best["sortino"] and calmar > current_best["calmar"]
                ):
                    best.setdefault(env_name, {})[algo_name] = {
                        "iter": idx, "sortino": sortino, "calmar": calmar
                    }

    return {env: {algo: v["iter"] for algo, v in algos.items()} for env, algos in best.items()}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| `@app.on_event("startup")` | `@asynccontextmanager lifespan` | FastAPI 0.93 (2023) | on_event deprecated; lifespan is the standard |
| Ollama /generate endpoint | Ollama /api/chat with format= schema | 2024 | Structured output support; /generate still works but /chat + format is cleaner |
| APScheduler BackgroundScheduler | APScheduler AsyncIOScheduler | APScheduler 3.x | AsyncIOScheduler integrates with asyncio event loop; BackgroundScheduler creates its own thread |
| httpx sync client | httpx async client (AsyncClient) | httpx 0.20+ | Async client required in async FastAPI routes to avoid blocking event loop |
| Docker CMD curl healthcheck | Docker CMD wget healthcheck (Ollama) | 2024 | Ollama image doesn't include curl; wget is available |

**Deprecated/outdated:**
- `APScheduler>=4.0`: Completely different API (async-first redesign). Project pins <4.0. Do not upgrade.
- Ollama nomic-embed-text: Explicitly excluded by project decision. Do not add embeddings endpoint.
- Relative imports in services/memory/: Use absolute imports within the package (e.g., `from memory.db import ...` with PYTHONPATH=/app/services/memory or `from .db import ...`).

## Open Questions

1. **Sortino from walk-forward vs. final training**
   - What we know: Walk-forward folds produce OOS metrics (sharpe, mdd, profit_factor). Final training on recent data does NOT run another WF evaluation — it trains and saves model.zip.
   - What's unclear: Where do per-iteration Sortino/Calmar values come from for best-model selection? The existing final_training dict only contains `timesteps`, `wall_clock_s`, `converged_at`, `model_path`. Sortino is NOT currently computed for final deployment models.
   - Recommendation: Planner should decide: (a) use WF fold OOS Sortino as the proxy for final model quality, or (b) add a lightweight single-fold backtest eval after final training to get Sortino/Calmar. Option (a) is simpler and consistent with existing logic.

2. **MemoryClient API key support**
   - What we know: Existing MemoryClient in src/swingrl/memory/client.py uses urllib without auth headers. The memory service now requires X-API-Key on all endpoints.
   - What's unclear: The MemoryClient needs to add `X-API-Key: {key}` to all requests.
   - Recommendation: Add `api_key: str = ""` param to MemoryClient.__init__(). Include header in all requests. Default empty = no auth (backward compatible with tests).

3. **Ollama resource stats for Discord embeds**
   - What we know: CONTEXT.md says "Ollama resource stats (RAM, tokens/sec, loaded models) included in all embeds." Two options: (a) parse `docker stats swingrl-ollama --no-stream --format json` from within the memory service, or (b) call Ollama's undocumented /api/ps endpoint which lists loaded models.
   - What's unclear: /api/ps availability is not in official Ollama docs (as of 2026-03-14).
   - Recommendation: Use `docker stats` via subprocess OR just skip RAM/tokens stats and report loaded model names via /api/tags. Claude's discretion applies here per CONTEXT.md.

4. **consolidation_quality table location**
   - What we know: CONTEXT.md specifies `consolidation_quality table` in DuckDB. But consolidations are stored in SQLite (memory.db). The quality log belongs next to what it's logging.
   - What's unclear: Should consolidation_quality be in DuckDB (market_data.ddb) or SQLite (memory.db)?
   - Recommendation: SQLite memory.db is the natural home — consolidation happens in the memory service, quality log is operational data not market data. Add to services/memory/db.py DDL.
