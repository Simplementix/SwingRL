**scripts/train_pipeline.py modifications:**

Add new CLI argument: `--iterations N` (default 0 = single baseline, no multi-iteration). When N > 0: run iteration 0 as baseline, then N iterations with memory enabled.

Add helper functions at module level:

1. `save_training_state(state: dict, path: Path) -> None`: Write to .tmp then os.replace (POSIX atomic). Per RESEARCH.md Pattern 5.

2. `load_training_state(path: Path) -> dict`: Load or return default {"completed_iterations": [], "current_iteration": 0, "iteration_results": {}}.

3. `select_best_per_algo_env(state: dict) -> dict[str, dict[str, int]]`: For each algo x env, find iteration with highest WF OOS Sortino. Calmar as tiebreak. Use WF fold OOS metrics as proxy (option a from RESEARCH.md Open Question 1 — simpler, consistent with existing logic). Compute Sortino from fold sharpe data: Sortino = mean(fold_sharpe_values) * sqrt(fold_count) as approximation, or use existing fold metrics if sortino is recorded. If neither exists, fall back to ensemble Sharpe as ranking metric.

4. `deploy_best_models(state: dict, models_dir: Path) -> dict`: For each algo x env, copy model.zip + vec_normalize.pkl from models/iterations/iter_{best}/ to models/active/. Use shutil.copy2. Return deployment map.

5. `write_comparison_report(state: dict, report_path: Path) -> None`: Write data/training_comparison.json with per-iteration, per-algo, per-env metrics. Include: iteration index, memory_enabled, ensemble_sharpe, ensemble_mdd, per-algo WF OOS metrics, wall_clock_s, and the winning iteration per algo x env.

6. `check_memory_service_health(base_url: str, timeout: float = 5.0) -> bool`: GET /health, return True if 200.

7. `wait_for_memory_service(base_url: str, poll_interval: float = 30.0) -> None`: Poll health endpoint every 30s indefinitely until healthy. Log warning on each failed poll. Send Discord alert on first failure (via existing webhook if configured), then every 15 min.

Modify `main()`:
- Parse --iterations argument
- If iterations > 0, call `run_all_iterations()` instead of the existing per-env loop
- run_all_iterations orchestrates:
  a. Apply DuckDB training telemetry DDL (from Plan 02's apply_training_telemetry_ddl)
  b. For each iteration 0..N:
     - Skip if in state["completed_iterations"]
     - Configure memory_agent.enabled based on iteration index
     - Override models_dir to models/iterations/iter_{i}/ so each iteration has isolated storage
     - Wait for memory service health if memory enabled
     - Call run_environment() for equity then crypto (existing function)
     - On algo failure: retry once, then skip (mark failed in state)
     - Save iteration results to state
     - Atomically write training_state.json
     - If memory enabled and i > 0: call MemoryClient.consolidate() between iterations
     - Send Discord alert for iteration start/end (no-op if webhook unconfigured)
  c. After all iterations: select_best_per_algo_env(), deploy_best_models(), write_comparison_report()
  d. Run ensemble gate on best iteration's models (check_ensemble_gate on best WF results)
  e. Write final training_report.json with iteration summary

IMPORTANT per RESEARCH.md Pitfall 7: Each iteration saves to models/iterations/iter_{N}/{env}/{algo}/ — NOT to models/active/. The existing _check_algo_checkpoint() checks models/active/ — so override models_dir per iteration to avoid false checkpoint hits. The existing `--force` flag still works within each iteration.

IMPORTANT per RESEARCH.md Pitfall 3: Baseline run (iteration 0) MUST write DuckDB training_runs rows. Verify _write_model_metadata() sets appropriate run_type. This counts toward cold-start threshold.
