  <behavior>
    - Test: VerificationResult and CheckResult dataclasses can be constructed and serialized to dict
    - Test: _check_equity_rows returns passed=False when DuckDB ohlcv_daily has no rows for a configured symbol
    - Test: _check_equity_rows returns passed=True when all configured equity symbols have rows (>100)
    - Test: _check_crypto_rows returns passed=False when ohlcv_4h is missing a configured crypto symbol
    - Test: _check_macro_series returns passed=False when a configured FRED series is missing from macro_features
    - Test: _check_obs_vector returns passed=False when observation vector contains NaN
    - Test: _check_obs_vector returns passed=True and validates shape (156-dim equity, 45-dim crypto) when vector is clean
    - Test: run_verification returns VerificationResult with overall passed=False if any check fails
    - Test: write_report writes valid JSON to the specified path
    - Test: print_summary outputs pass/fail per check to stdout (capture with capsys)
  </behavior>
  <action>
    Create `src/swingrl/data/verification.py` with:

    1. `VerificationResult` and `CheckResult` dataclasses:
       ```python
       @dataclass
       class CheckResult:
           name: str
           passed: bool
           detail: str

       @dataclass
       class VerificationResult:
           passed: bool
           checks: list[CheckResult]
       ```

    2. Individual check functions (all take DuckDB cursor + SwingRLConfig):
       - `_check_equity_rows(cursor, config)` — `SELECT symbol, COUNT(*) FROM ohlcv_daily GROUP BY symbol`, verify all `config.equity.symbols` present with >100 rows each
       - `_check_crypto_rows(cursor, config)` — same for `ohlcv_4h` and `config.crypto.symbols`
       - `_check_macro_series(cursor, config)` — `SELECT DISTINCT series_id FROM macro_features`, verify all `config.fred.series` present
       - `_check_equity_date_range(cursor, config)` — `SELECT MIN(date), MAX(date) FROM ohlcv_daily`, log coverage
       - `_check_crypto_date_range(cursor, config)` — same for `ohlcv_4h` with datetime column
       - `_check_obs_vector(pipeline, environment, date_str)` — call `pipeline.get_observation(environment, date_str)`, assert shape matches expected dim and `np.isnan(obs).sum() == 0`. Expected dims: equity=156, crypto=45. Use the LAST available date (query `MAX(date)` from relevant feature table).

    3. `run_verification(config: SwingRLConfig) -> VerificationResult`:
       - Create DatabaseManager, open DuckDB cursor
       - Run all checks, collect CheckResults
       - Create FeaturePipeline with the cursor for obs vector checks
       - Return VerificationResult with `passed = all(c.passed for c in checks)`

    4. `write_report(result: VerificationResult, path: Path) -> None`:
       - `path.parent.mkdir(parents=True, exist_ok=True)`
       - Write `json.dumps(dataclasses.asdict(result), indent=2, default=str)`

    5. `print_summary(result: VerificationResult) -> None`:
       - Print each check: `[PASS]` or `[FAIL]` + name + detail
       - Print overall result at end

    Follow all CLAUDE.md conventions: `from __future__ import annotations`, structlog logging, type hints, docstrings.

    Create `tests/data/test_verification.py` with tests for all behaviors listed above. Use an in-memory or tmp-path DuckDB for tests. Mock `FeaturePipeline.get_observation()` to return controlled numpy arrays. Call `DatabaseManager.reset()` in teardown. Use `tmp_config` fixture from conftest.py.

    For observation vector tests, mock `FeaturePipeline` rather than constructing real feature data — the verification module's job is to CHECK the output, not produce it.
