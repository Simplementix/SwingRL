   - REWRITE `duckdb()` context manager to open a fresh connection and close on exit:
     ```python
     @contextlib.contextmanager
     def duckdb(self) -> Generator[Any, None, None]:
         """Open a fresh DuckDB connection per call, close on exit.

         No persistent connection is held. The DuckDB write lock is held only
         for the duration of the context block (milliseconds for reads, seconds
         for writes). This enables concurrent subprocess access from retrain jobs.
         """
         self._duckdb_path.parent.mkdir(parents=True, exist_ok=True)
         conn = duckdb.connect(str(self._duckdb_path))
         try:
             yield conn
         finally:
             conn.close()
     ```
   - UPDATE `close()` method: remove the `if self._duckdb_conn is not None` block entirely. Only log "database_manager_closed" (or remove method body to pass). Keep the method signature for API compatibility.
   - UPDATE `reset()`: the `cls._instance.close()` call still works since close() is still a valid method.
   - UPDATE class docstring: change "DuckDB uses a persistent connection with per-call cursors" to "DuckDB opens a fresh connection per context manager call (no persistent state)"
   - NOTE: The `duckdb()` method now yields a CONNECTION (not a cursor). All callers already use `with db.duckdb() as cursor:` where `cursor` is actually a connection that supports `.execute()`. DuckDB connections and cursors share the same execute API, so no caller changes needed.
